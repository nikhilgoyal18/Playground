### How to Ship Your First Pull Request as a PM `Product`
- Matt MacInnis (CPO at Rippling, $16B+ workforce platform) publicly stated PMs at Rippling are now shipping code — framing it as three structural shifts already in motion at one of the largest product orgs in tech
- Garry Tan (CEO, Y Combinator) open-sourced his full Claude Code setup ("gstack") — hit 33K GitHub stars in a week, now past 65K — and is personally shipping 100 PRs/week while running YC
- Boris Cherny (creator of Claude Code at Anthropic) ships 20–30 PRs a day, 100% AI-written, and has uninstalled his IDE entirely — signals that AI-native workflows are displacing traditional dev tooling even for the tool's own creator
- OpenAI's Codex team shipped an entire product with 1,500 merged PRs and zero manually written code; Superhuman uses Codex so PMs can "contribute lightweight code changes without pulling in an engineer, except for code review"
- The new role boundary: PM zone expands into copy, config, AI prompts, planning docs in git, small front-end changes, and production monitoring — Engineering concentrates on architecture, security, complex logic, and code review (the high-judgment problems)
- Key insight from a startup CTO: "When PMs build their own ideas, their specifications get sharper, because they now understand what the agent needs to execute well. Sharper specs produce better agent output" — shipping makes every other PM skill better, not redundant
- Aakash frames this as additive, not role-replacing: "The cost of building the wrong thing just dropped to near zero. The PM who ships has tighter feedback loops on everything"
