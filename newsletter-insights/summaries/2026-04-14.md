## Paweł from The Product Compass

### Your .claude/ Folder Is a Production Agent `AI/ML`
- The `.claude/` folder (CLAUDE.md, skills, MCPs, hooks, sub-agents) is the identical deployable artifact in dev and production — no translation layer; the same CLAUDE.md that shapes Claude Code in your terminal loads as the system prompt of your production agent
- Skills (`.claude/skills/`) are reusable behaviors in markdown — same file format and same loader work identically in dev and prod; changing agent behavior is editing a markdown file, not a sprint or redeploy cycle
- Hooks replace n8n-style IF-node-per-tool workflows in five lines of code; `PostToolUse` can log everything the agent read (cost, duration, files read) for live UI display; OTEL exports full traces to Langfuse or any OTLP backend for dashboards
- Sub-agents live at `.claude/agents/<name>/` — each directory is a self-contained agent with its own CLAUDE.md, skills, MCP config, and docs; adding a second agent to your product is creating a new folder
- MCPs (`.claude/mcp.json`) extend the agent's tool surface — dev config talks to Chrome/filesystem, prod config talks to your search API and database; different servers, identical contract
- You can skip Claude Managed Agents entirely for request-response agents inside an app you already host — the Agent SDK is a smaller dependency than a full hosted platform
- Key gotcha: use a postbuild trick to stop production from inheriting your repo's dev CLAUDE.md — otherwise prod behavior leaks dev-only context
- For PMs: iterate on AI agent behavior at the speed of editing a Notion doc; no code review or redeploy required when changing the agent's instructions

## ByteByteGo
*1 issue this period*

### Figma Design to Code, Code to Design: Clearly Explained `Engineering`
- Two naive approaches both fail: (1) screenshots give the LLM a visual reference but no precise values — it guesses spacing as 20px instead of the exact 24px; (2) raw Figma REST API JSON dumps thousands of lines including blend modes, export settings, and pixel coordinates that drown the LLM in noise and degrade output quality
- Figma's MCP server (launched June 2025) is the middle ground: filters raw API data, converts pixel positions to layout relationships (e.g. "centered inside its parent"), maps raw hex colors to design token references, and flattens nested layers to match what a developer would actually build
- Design-to-code workflow: developer selects a Figma frame, copies the URL, pastes it into Claude Code or Codex with "Implement this design" → agent discovers tools (`get_design_context`, `get_screenshot`, `get_metadata`), calls them with the frame URL, receives clean structured context, generates matching code
- Source interviews: Emil Sjölander, Aditya Muttur, and Shannon Toliver from the Figma engineering team behind the MCP server
- New in 2026: two additional workflows — (1) generate designs FROM coding tools like Claude Code and Codex, and (2) agents writing directly back into Figma designs (code-to-design); the bridge is now bidirectional
- Engineering challenges remain: complex component states and responsive behavior are not yet fully solvable; the MCP server reduces noise but can't fully model dynamic interaction patterns
- MCP (Model Context Protocol) is the standard defining how AI agents discover and call external tools; Figma's implementation is one of the most mature production examples of MCP being used to bridge a non-text knowledge domain (visual design) into LLM-readable context

## Jay Stansell — Product Coalition
*1 issue this period*

### When 'Viable' Stopped Being Enough `Product`
- **2015–2019 era**: Fabrice Wegner (Product Coalition, Dec 2015) captured the gospel — "Is it wrong to want to ship fast?" The answer was no; Vaibhav Gupta's 2019 MVP guide cited the classic Uber example (Travis Kalanick driving a car, a basic web app) and the Build-Measure-Learn loop as the canonical model
- The key line teams consistently missed from Eric Ries: "Your customers won't get happier or pay you a dollar if your team has learnt something" — learning was meant to justify shipping the minimum, but teams took that as a license for minimum effort
- **2020 split**: Yellow Systems (March 2020) drew a new line — "a reduction in features is expected, but there should be no reduction in quality"; Carlos Gonzalez de Villaumbrosia coined the Minimum Lovable Product (MLP), arguing "being Viable just doesn't cut it anymore. You need to be Lovable"
- The MLP argument rested on a real market shift: by 2020, users had Slack, Notion, and Figma as their baseline for "acceptable first version" — the floor had moved up and most product teams hadn't noticed
- **2025 backlash**: Garima Srivastava (Product Coalition) declared the MVP dead — "half-baked products that nobody loves"; her analogy: serving an uncooked pizza and asking customers to imagine how great it'll be when finished; viability is no longer the bar, loveability is
- Jay Stansell's conclusion: the original Ries MVP was always a *learning tool*, not a product launch strategy; the PM community misread "minimum viable" as "minimum effort" and spent a decade shipping products that launched to crickets — the correction is overdue but the MVP concept itself is not wrong, just chronically misapplied

## Aakash Gupta — Product Growth

### How PMs Can Take Advantage of Karpathy's Second Brain `AI/ML`
- Karpathy's system (19.7M views on X, 5,000+ GitHub stars) solves the PM research black hole: instead of retrieving at query time (what RAG, NotebookLM, ChatGPT uploads all do), the LLM compiles and maintains the knowledge base AT INGEST — "the cross-references are already there, the contradictions have already been flagged, the synthesis already reflects everything you've read"
- Root cause of failed PM knowledge bases: maintenance was assigned to the wrong person — PMs managing 6 workstreams simultaneously can't keep a wiki current; falling behind by one week creates enough guilt to ensure it never gets reopened; the LLM doesn't get bored, making it structurally better at the bookkeeping job
- Three components: **Raw** (junk drawer — dump interview transcripts, competitor changelogs, analyst reports without organizing; that's the AI's job); **Wiki** (AI-built and AI-maintained synthesis: summaries, concept pages, entity pages for competitors/stakeholders/user segments — the human never writes the wiki); **Schema** (single CLAUDE.md defining categories and contradiction-handling rules — written once, followed every session)
- PM-specific mapping: Raw = your working Notion folder that actually works; Wiki = the synthesis that used to live in your head and vanish when the project ended; Schema = the instruction that makes the AI a disciplined colleague rather than a chatbot that forgets you exist
- Aakash has been running for 1.5 weeks and reports week three feels qualitatively different as the wiki develops critical mass — early days it's a dump; by week three it starts surfacing connections you'd forgotten you made
- Two downloadable assets provided: a 60-second setup skill file, and a PM-tailored CLAUDE.md — the skill handles the initial scaffold so you don't have to understand every piece before starting
- Real-world adoption: Farza (founder of buildspace) took the same pattern and built Farzapedia — a personal knowledge base that compounds across sessions rather than resetting
- Honest limitation noted: the system requires Claude Code; the payoff accrues slowly in weeks 1–2 before becoming clearly worthwhile; not a day-one win
