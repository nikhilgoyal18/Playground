## 1. Your .claude/ Folder Is a Production Agent
**From:** Paweł from The Product Compass
**Date:** April 14, 2026

Paweł ran a spike using the same Claude Code primitives he uses daily — CLAUDE.md, skills, MCPs, hooks — and shipped a working Knowledge Chatbot in an afternoon pointed at a Next.js app. The core insight: your `.claude/` folder is not just a dev tool, it is the deployable production runtime.

**Key takeaways:**
- The same CLAUDE.md that guides Claude Code in your terminal loads as the system prompt of your production agent — no translation layer between dev and prod.
- Skills (`.claude/skills/`), MCP configs, hooks, and sub-agents are all version-controlled markdown files that work identically in dev and production.
- Hooks replace n8n-style IF-node-per-tool workflows in five lines of code, and provide live per-response observability (cost, duration, files read) without a new vendor.
- For PMs: changing agent behavior is editing a markdown file, not a sprint or a redeploy cycle.
- The Agent SDK is sufficient for request-response agents inside an app you already host — no need for Claude Managed Agents unless you need a hosted runtime.

## 2. Figma Design to Code, Code to Design: Clearly Explained
**From:** ByteByteGo
**Date:** April 14, 2026

A deep technical breakdown of how Figma's MCP server bridges the longstanding gap between design files and production code, based on interviews with the Figma engineering team.

**Key takeaways:**
- The two naive approaches both fail: screenshots lack precision; raw Figma API JSON is too noisy and token-inefficient for LLMs to act on reliably.
- Figma's MCP server acts as a middle ground — it filters the raw API data, converts pixel positions to layout relationships, maps raw colors to design tokens, and flattens nested layers into developer-friendly structure.
- The design-to-code workflow integrates with agents like Claude Code and Codex: the agent discovers tools (e.g. `get_design_context`, `get_screenshot`), calls them with a Figma frame URL, and receives clean structured context to generate matching code.
- Figma also launched a code-to-design workflow, enabling agents to write directly back to Figma designs.
- Engineering challenges in closing the design-to-code gap remain, particularly around complex component states and responsive behavior.

## 4. When 'Viable' Stopped Being Enough
**From:** Jay Stansell (Product Coalition)
**Date:** April 14, 2026

A retrospective on how the PM community's thinking around the MVP evolved dramatically over a decade, traced through Product Coalition's own archive.

**Key takeaways:**
- **2015–2019:** The dominant message was "just ship something." Speed was gospel, with learning used to justify rough quality — a permission many teams abused.
- **2020:** A split emerged. One camp maintained that launch delays kill startups; another introduced the Minimum Lovable Product (MLP), arguing that by 2020 users had been trained on tools like Slack and Notion and would no longer accept "merely viable."
- **2025:** A contributor declared the MVP dead, comparing shipping a half-baked product to serving an uncooked pizza and asking customers to imagine how good it will be.
- **Jay's take:** The original MVP concept (Eric Ries) was always a *learning tool*, not a license for low-effort launches. The community misread "minimum viable" as "minimum effort" and spent a decade paying for it. The correction is overdue — but the MVP is not dead, just widely misapplied.

## 5. How PMs Can Take Advantage of Karpathy's Second Brain
**From:** Aakash Gupta (Product Growth)
**Date:** April 13, 2026

Aakash adapts Andrej Karpathy's viral "Second Brain" system (19.7M views) specifically for PM workflows, addressing the core problem that PM research resets every quarter.

**Key takeaways:**
- The root failure of every PM knowledge base: maintenance was assigned to the wrong person. PMs managing six workstreams simultaneously cannot keep a wiki current, and guilt from falling behind ensures it gets abandoned.
- Tools like NotebookLM and ChatGPT uploads retrieve at query time — they never build persistent, compounding understanding between sessions.
- Karpathy's fix: have the LLM compile and maintain the knowledge base *at ingest*, not at query time. Three components: **Raw** (your unorganized dump), **Wiki** (AI-built synthesis, summaries, entity pages), and **Schema** (a single CLAUDE.md that defines structure and contradiction-handling rules — written once).
- For PMs: Raw is your working Notion folder; Wiki is the synthesis that used to vanish when a project ended; Schema is what makes the AI a disciplined colleague rather than a forgetful chatbot.
- Aakash has been running the system for 1.5 weeks and provides a downloadable skill file and PM-tailored CLAUDE.md for setup.
